# Домашнее задание: тестирование кейс-промптов на реальном проектном контексте Kitchen Helper

## 1. Цель работы

Цель работы — протестировать три кейс-промпта из урока и оценить, как они помогают в развитии реального учебного проекта Kitchen Helper.

В работе проверяются три типа промптов: Summary Prompt, Code Structure Prompt и Task Planning Prompt. Каждый из них используется не как отдельное абстрактное упражнение, а как способ получить полезный артефакт для дальнейшей работы над проектом.

## 2. Почему выбран Kitchen Helper

Kitchen Helper выбран потому, что это не случайная тема, а учебный проект, который уже имеет практическую цель. Это Telegram-бот-помощник по домашней готовке: пользователь пишет, какие продукты есть дома, или задаёт бытовой вопрос о готовке, а бот помогает придумать простой рецепт или кулинарную идею, поддерживает контекст диалога и позволяет сбросить его через команду /reset.

Проект можно дальше развивать до небольшой beta-версии для реального пользователя. Поэтому три кейс-промпта проверяются как части одного рабочего процесса: product brief → architecture proposal → beta roadmap.

Такой подход показывает не повторение примеров преподавателя, а применение метода из урока к собственному проекту. Один и тот же контекст проходит через разные стадии: описание смысла проекта, проектирование структуры и планирование ближайшей разработки.

## 3. Инструмент и параметры

- Python script from Cursor terminal
- gpt-4o-mini
- temperature=0.2
- Markdown output
- три JSON-промпта по структуре урока

## 4. Summary Prompt → Product Brief

Что тестировалось: способность Summary Prompt быстро превратить описание проекта в понятный product brief.

Какой артефакт получен: `outputs/01_product_brief.md`. Это краткий документ по Kitchen Helper: цель проекта, пользователь, текущие функции, ограничения и возможное развитие.

Чем полезен для проекта: product brief помогает зафиксировать, что именно делает бот и для кого он создаётся. Его можно использовать как основу для README, описания проекта, презентации или дальнейшего планирования.

Что можно улучшить в промпте: можно добавить ограничение по длине и попросить модель отдельно выделять не только текущие функции, но и границы первой beta-версии.

## 5. Code Structure Prompt → Architecture Proposal

Что тестировалось: способность Code Structure Prompt предложить практичную структуру Python Telegram-бота, которую можно использовать для развития Kitchen Helper.

Какую структуру предложила модель: в `outputs/02_architecture_proposal.md` получена простая структура Python-проекта с `bot.py`, `config.py`, папками `handlers/`, `services/`, `models/`, `tests/`, а также файлами `.env`, `requirements.txt` и `README.md`. В предложении отдельно описаны обработчики, сервис генерации рецептов, сервис контекста, модель пользовательского контекста, тесты и конфигурация.

Что полезно: такой результат помогает перейти от идеи к организации кода. Особенно полезны разделение ответственности, отдельная зона для конфигурации, место для тестов и заранее предусмотренная точка расширения под избранные рецепты.

Что нужно критически фильтровать: модель может предложить слишком много слоёв, сложное хранилище или архитектуру “на вырост”. Для Kitchen Helper это нужно сокращать: проект маленький, его делает один разработчик, поэтому важнее простая структура, которую реально поддерживать.

## 6. Task Planning Prompt → Beta Roadmap

Что тестировалось: способность Task Planning Prompt превратить цель первой beta-проверки в реалистичный план на 1-2 недели.

Какой план получен: в `outputs/03_beta_roadmap.md` получен roadmap с этапами по основному функционалу, UX и тестированию. Задачи разделены на Must/Should/Nice, отдельно указаны риски, критерии готовности, минимальный план тестирования и первые задачи для старта.

Чем полезен для подготовки beta-версии: roadmap помогает понять, какие задачи действительно важны перед первым реальным пользователем. Он удерживает фокус на UX, стабильности, понятном рецепте или идее блюда, сбросе контекста и простом сохранении понравившегося варианта.

Какие задачи стоит взять в первую очередь: проверить основной сценарий “продукты → рецепт или идея блюда”, улучшить сообщения бота, убедиться, что /reset работает понятно, добавить простой механизм избранного и подготовить минимальные тесты для ключевых бытовых кулинарных запросов.

## 7. Сравнение трёх промптов

| Промпт | Полученный артефакт | Практическая польза | Ограничения результата |
|--------|-------------------|----------------------|-----------------------|
| Summary Prompt | Product Brief | Быстро формулирует суть проекта, пользователя, функции и ограничения | Может быть слишком общим без лимита по длине и акцента на beta-границы |
| Code Structure Prompt | Architecture Proposal | Даёт черновик структуры проекта и разделение ответственности | Нужно фильтровать лишнюю сложность и не принимать “архитектуру на вырост” без проверки |
| Task Planning Prompt | Beta Roadmap | Разбивает развитие на этапы, приоритеты, риски и критерии готовности | Требует ручной оценки сроков и выбора задач, которые реально успеть одному разработчику |

## 8. Общий вывод

Кейс-промпты полезны как повторяемые рабочие шаблоны. Они помогают не формулировать задачу с нуля каждый раз, а направлять модель на конкретный тип результата.

На примере Kitchen Helper видно, что Summary Prompt помогает быстро сформулировать суть проекта, Code Structure Prompt — получить черновик архитектуры, а Task Planning Prompt — разложить развитие на конкретные шаги. Наиболее полезными для Kitchen Helper оказались Code Structure и Task Planning, потому что они напрямую помогают двигаться к beta-версии: понять, как организовать код и какие задачи делать в первую очередь.
